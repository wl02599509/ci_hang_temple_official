# Skills 維護規範

這份文件定義 Good UX Skills 在 `.agents/skills` 目錄下的共用維護標準。

適用對象：

- 維護 skill 內容的人
- 新增 skill 的貢獻者
- 會讀取、整理、擴充 `SKILL.md`、`TEMPLATE.md`、`examples.yaml` 的 agent

---

## 0. 技能架構總覽

Good UX Skills 採用**設計思考五大流程**作為核心技能架構：

### 核心技能（5 個）

位於 `.agents/skills/` 根目錄，為主要安裝與使用的技能：

1. **empathize** - 同理洞察（整合訪談與人物誌）
2. **define** - 問題定義（整合資訊架構與 a11y 限制）
3. **ideate** - 發想構思（整合線框與視覺風格，含 20 種 UI 風格參考）
4. **prototype** - 原型製作
5. **test** - 可用性驗證（整合可用性測試與 a11y 驗證）

### References 參考資料（11 個）

位於 `.agents/skills/references/`，包含：

**詳細方法（9 個）**：
- `user-interview`, `persona-creation`, `information-architecture`, `wireframing`, `ui-visual-design`, `usability-testing`, `accessibility-design`, `design-system`, `prototyping`

**使用指引（2 個）**：
- `SKILL_USAGE_GUIDE.md` - Skill 使用規則、避免錯亂、專案檢測、開發選項
- `KEYWORDS_LIBRARY.md` - 五大技能的觸發關鍵字與衝突解決規則

**References 管理原則**：
- References 不再透過 CLI 安裝，僅供查閱與引用
- 核心技能的 SKILL.md 在「參考資料 (References)」段落標示對應的 references 檔案
- 新增核心技能內容時，優先整合至五大流程，避免建立新的 references
- 保持 references 簡潔，避免過度複雜的文件
- 核心技能的 References 段落需與 `references/` 目錄內容保持同步
- 新增核心技能內容時，若涉及 references 引用，需同步更新引用說明

## 1. 核心原則

所有 skill 的內容維護都必須遵守以下順序：

1. **先閱讀 `SKILL.md`**
   - 理解技能目標、觸發條件、輸出責任與邊界

2. **再閱讀 `TEMPLATE.md`**
   - 確認資料格式、欄位規則、最小必要欄位、品質標準與驗證清單

3. **最後閱讀 `examples.yaml`**
   - 取得可直接引用的內容、案例、模式、分類與命名方式

### 強制規則

- **不得跳過 `TEMPLATE.md` 直接新增 YAML 資料**
- **不得只根據 `examples.yaml` 自行推斷新格式**
- **最終輸出格式必須優先對齊 `TEMPLATE.md`**
- **內容細節才從 `examples.yaml` 補足**

---

## 2. 三種檔案的角色分工

### `SKILL.md`

用途：定義 skill 的行為規則。

負責內容：

- 任務定義
- 何時使用
- 必要輸入
- 預期輸出
- 不適用情境
- 觸發條件
- 必要澄清
- 執行步驟
- 執行檢查
- 範例資料庫說明
- 資料使用規則

**不應承擔的內容**：

- 大量結構化案例資料
- 複雜 YAML 格式規範
- 長篇資料庫內容本體

### `TEMPLATE.md`

用途：定義新增與維護資料時必須遵守的格式規範。

負責內容：

- YAML 結構
- 欄位規則
- 最小必要欄位
- 高品質標準
- 驗證清單
- 維護原則

**不應承擔的內容**：

- 大量實際案例資料
- skill 的使用情境說明
- 與 `SKILL.md` 重複的任務定義

### `examples.yaml`

用途：提供 skill 可直接引用的內容資料庫。

負責內容：

- 題庫
- 範本
- 模式
- 清單
- 案例
- 規則資料
- 可檢索的分類與標籤

**不應承擔的內容**：

- 長篇說明文字
- 教學型文件段落
- 超出 skill 職責邊界的資料

---

## 3. 每個 skill 的標準結構

每個 skill 目錄應維持以下三件組：

```text
skill-name/
├── SKILL.md
├── TEMPLATE.md
└── examples.yaml
```

### 例外

- `ideate` 使用 `styleprompts.yaml` 作為 UI 風格資料檔（含 20 種視覺風格參考）
- 但仍必須遵守：
  - 先讀 `SKILL.md`
  - 再讀 `TEMPLATE.md`
  - 最後讀 YAML 資料檔

---

## 4. 新增資料時的判斷流程

新增任何資料前，先依照以下順序判斷：

### Step 1：確認是否真的需要新增

先問：

- 現有 `examples.yaml` 是否已經有相近項目？
- 只是名稱不同，還是本質真的不同？
- 是新的模式，還是舊模式的變體？

如果只是細微差異：

- 優先補到既有項目
- 或補到既有項目的 `variations`、`tags`、`best_for` 等欄位

### Step 2：確認新增格式

新增前必須先對照 `TEMPLATE.md`：

- 欄位是否符合模板
- 命名是否符合規則
- 是否具備最小必要欄位
- 是否能通過驗證清單

### Step 3：確認新增價值

新增項目必須至少符合其中一項：

- 補足目前沒有涵蓋的高頻場景
- 提供明確不同的模式或案例
- 提升 skill 的檢索精準度
- 補足現有資料無法支援的決策情境

如果做不到以上任一項，**不要新增**。

---

## 5. 命名規範

### YAML key

- 一律使用英文 `snake_case`
- 例如：
  - `pain_points`
  - `decision_making`
  - `fidelity_levels`
  - `card_sorting_guide`

### ID 規範

若資料結構有 `id`，必須穩定且唯一。

建議格式：

- 問題：`q01`、`q02`
- 風格：`s01`、`s02`
- 檢查項：短而穩定的英文識別，例如 `keyboard_access`

### 名稱規範

- 給人看的 `name` 使用正體中文
- 給系統檢索的 key 使用英文 snake_case
- 同一個 skill 中，不可混用多套命名邏輯

### 標籤規範

- 標籤要精準，不要過多
- 建議每筆資料 2-5 個 tag 即可
- 優先使用可檢索、可分類、可匹配的詞

---

## 6. 品質標準

所有資料都應符合以下標準：

### 精簡

- 資料是給 skill 檢索與展開，不是寫百科全書
- 每筆內容保留最有辨識度與決策價值的資訊

### 精準

- 每個欄位都要能回答一個清楚問題
- 避免空泛描述，例如：
  - 不佳：現代、好看、專業
  - 較佳：中性色搭配單一品牌主色，適合 SaaS 首頁與儀表板

### 可重用

- 內容應能跨多個任務重複使用
- 避免只服務單一專案特例

### 可檢索

- key、tag、分類應幫助 skill 做匹配
- 不要只寫給人看，卻無法被有效引用

### 不重複

- 相似模式不要拆成很多近似項目
- 若只是輕微差異，應合併到既有結構

---

## 7. 修改 `SKILL.md` 的原則

可以修改 `SKILL.md` 的情況：

- skill 的定位不清楚
- 觸發條件不足
- 輸出格式不夠穩定
- 缺少資料使用規則
- 新增了必要的澄清問題或執行步驟

不應該修改 `SKILL.md` 的情況：

- 只是想新增案例資料
- 只是想補一筆 YAML 項目
- 只是想調整細部命名

這些應優先改在：

- `TEMPLATE.md`
- `examples.yaml`

---

## 8. 修改 `TEMPLATE.md` 的原則

可以修改 `TEMPLATE.md` 的情況：

- 現有欄位定義不夠清楚
- 缺少最小必要欄位
- 驗證清單不足
- 品質標準不夠具體
- 新的資料類型需要正式納入模板

修改 `TEMPLATE.md` 時必須注意：

- 不要頻繁改動核心欄位結構
- 不要讓模板比資料本體更複雜
- 不要加入難以落地的規定

如果變動會影響既有 `examples.yaml`，應先評估：

- 是否要同步調整既有資料
- 是否需要保留舊格式相容性

---

## 9. 修改 `examples.yaml` 的原則

修改 `examples.yaml` 時必須遵守：

- 優先重用既有分類
- 優先補充既有項目，而不是新增相似項目
- 新增欄位前先確認 `TEMPLATE.md` 是否允許
- 新增後必須仍能被 skill 直接引用

### 不應發生的情況

- 同一 skill 同時存在兩套欄位結構
- 相似內容用不同 key 命名
- 同一概念有多個重複 tag
- 加入大量不會被引用的內容

---

## 10. 新增一個新 skill 的最小流程

新增 skill 時，請依照以下順序：

1. 建立資料夾：`new-skill/`
2. 建立 `SKILL.md`
3. 建立 `TEMPLATE.md`
4. 建立 `examples.yaml`
5. 在 `SKILL.md` 中補上：
   - 範例資料庫
   - 資料使用規則
6. 檢查命名、欄位與內容是否一致
7. 確認該 skill 與既有 skill 的邊界清楚

---

## 11. 新增或修改資料前的檢查清單

### 新增前

- [ ] 已閱讀對應 `SKILL.md`
- [ ] 已閱讀對應 `TEMPLATE.md`
- [ ] 已檢查現有 YAML 是否已有近似項目
- [ ] 已確認新增內容有實際價值

### 新增後

- [ ] 欄位結構符合 `TEMPLATE.md`
- [ ] 命名符合規範
- [ ] 沒有重複 key 或重複 id
- [ ] 內容可被 skill 直接引用
- [ ] 資料足夠精簡且精準

---

## 12. 給 agent 的執行規則

如果你是讀取這份文件的 agent，請遵守：

1. 不要跳過 `TEMPLATE.md`
2. 不要直接憑空發明新的 YAML 結構
3. 不要在沒有檢查既有資料的情況下新增項目
4. 不要把 `examples.yaml` 當成最終輸出原文直接貼給使用者
5. 要先用模板約束格式，再用資料庫補內容
6. 若資訊不足，優先補澄清，不要硬生成低品質內容

---

## 12.1 禁止規則

以下情況視為明確違規，應直接避免：

- 禁止新增與既有項目只有字面差異、但本質相同的分類
- 禁止新增不符合 `TEMPLATE.md` 最小必要欄位的資料
- 禁止在沒有明確使用情境時擴充長尾資料
- 禁止把同一概念拆成多個近義 tag
- 禁止用中文、英文、縮寫混搭成不一致的 key 命名
- 禁止把專案特例直接寫成通用規則
- 禁止把暫時性的描述、討論草稿或未驗證假設寫進正式 YAML
- 禁止為了看起來完整而填入不會被 skill 使用的欄位

## 12.2 審查準則

每次新增或修改資料後，至少要用以下標準自我審查一次：

- **結構審查**：欄位是否完整且符合 `TEMPLATE.md`
- **命名審查**：key、id、tag 是否一致且可檢索
- **去重審查**：是否與既有項目高度重複
- **邊界審查**：內容是否仍屬於該 skill 的職責範圍
- **可用性審查**：skill 是否真的能引用這筆資料來改善輸出品質
- **精簡度審查**：是否存在冗詞、重複描述、過多標籤或無效欄位

---

## 13. 維護優先順序

當時間有限時，優先順序如下：

1. 修正結構不一致
2. 移除重複與模糊資料
3. 補足高頻使用場景
4. 提升標籤與分類精準度
5. 再考慮擴充長尾案例

---

## 14. 總結

Good UX Skills 的資料維護原則很簡單：

- **`SKILL.md` 管行為**
- **`TEMPLATE.md` 管格式**
- **YAML 管內容**

所有新增與修改都應遵守：

- **先看 skill 邊界**
- **再看模板規則**
- **最後才動資料**

只要持續遵守這個順序，skills 的品質、一致性與可維護性就能穩定擴張。
