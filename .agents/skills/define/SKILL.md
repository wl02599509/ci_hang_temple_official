# Define｜問題定義

## 任務定義
將研究洞察轉為明確問題敘述，對齊範疇、限制與成功指標，為後續 Ideate/Prototype 奠基。當你需要「收斂問題焦點」、「設定成功指標」、「定義 MVP 範圍」時，就應該使用此技能進行問題定義與需求優先排序。

## 何時使用
- 完成研究後需要收斂關鍵問題與需求
- 多方利害關係人需要共識與優先排序
- 需要定義成功指標與驗收標準（含 a11y/法規）
- 需要界定 MVP 範圍並設定產品開發優先順序

## 必要輸入
- Empathize 階段的洞察、人物誌、旅程
- 已知限制（技術/法規/可及性）與假設
- 商業目標與成功指標期望

## 預期輸出
- 問題敘述（POV/How Might We）
- 需求清單與優先順序（影響力 × 可行性）
- 成功指標/驗收標準（如完成率、錯誤率、WCAG 等級）
- MVP 範圍與假設列表

## 完成條件
- 關鍵問題有清楚的敘述與依據
- 限制條件與假設被列出並對齊
- 成功指標明確且可量測（含 a11y 要求）

## 不適用情境
- 尚未進行任何使用者洞察收集（建議回到 Empathize）
- 已確定解法，不需再討論問題定義

## 觸發條件

**完整關鍵字庫請參考**：`KEYWORDS_LIBRARY.md` > Define 章節

### 核心關鍵字（高優先級）
- **主要動詞**：定義問題、界定問題、收斂問題、設定目標、設定指標、排序、優先排序、界定範圍、定義範疇、確定 MVP
- **名詞**：問題定義、POV、HMW、需求收斂、優先順序、KPI、成功指標、驗收標準、MVP、範疇、限制條件、無障礙要求、a11y 限制、WCAG
- **情境描述**：如何量測成功、需要符合哪些標準、哪些功能優先做、如何界定專案範圍

### 前置條件
- 需要已有使用者洞察或研究資料（來自 Empathize）

### 排除條件（優先考慮其他技能）
- 若同時提到「還不知道使用者需求」→ 轉 Empathize
- 若同時提到「設計畫面/介面」→ 轉 Ideate
- 若同時提到「製作可點擊原型」→ 轉 Prototype

## 必要澄清

### 專案狀態檢查
在開始問題定義前，請先確認：
- 是否已完成使用者研究（Empathize）？
- 是否已有使用者洞察、人物誌或痛點清單？
- 如果沒有，建議先使用 `/empathize` 進行研究

### 問題收斂
- 主要使用者與情境是否一致？
- 技術/法規/可及性有哪些硬性限制？
- 成功指標如何量測？是否有基準值？
- 是否需要分 MVP 與後續版本？

### 下一步規劃
定義問題後，您想要：

**選項 A：用 React + Shadcn/UI 做原型**
- 完成設計後直接用程式碼實作可互動的原型
- 優點：真實互動、可直接測試、技術驗證
- 我會協助您建立完整的開發環境與元件

**選項 B：先產生規格文件（spec.md）**
- 產生詳細的設計規格文件，包含問題定義、成功指標、功能需求
- 優點：先整理思路、方便團隊溝通、之後再決定實作方式
- 我會產生結構化的 spec.md 檔案

### 避免跳過此階段
即使專案很急，也不建議跳過問題定義直接設計，因為：
- ❌ 沒有明確目標，設計方向容易偏離
- ❌ 沒有成功指標，無法驗證設計是否有效
- ❌ 沒有優先順序，容易做出過度設計

## 可搭配技能
- Empathize：引用洞察與人物誌
- Ideate：根據問題敘述發散解法
- Test：將成功指標轉為驗證任務

## 執行步驟
1. 彙整洞察：聚合痛點/目標，繪製問題陳述與 POV。
2. 界定範疇：角色、場景、限制（技術/法規/可及性）。
3. 優先排序：以影響力 × 可行性排列需求，確定 MVP。
4. 設定指標：定義成效衡量與驗收標準（含 a11y）。

## 執行檢查
- [ ] 問題敘述清晰、可追溯回洞察
- [ ] 限制/假設被列出並已共識
- [ ] 成功指標可量測且與目標一致
- [ ] MVP 範圍明確，已標記高優先需求

## 精簡範例輸出
```markdown
# Define 摘要
- 問題敘述：忙碌店主在結帳流程中因物流/付款選項複雜而放棄購物
- 限制：需支援行動端、WCAG AA 對比、現有金流 API
- 指標：結帳完成率 +15%，錯誤率 -20%，流程時間 -25%
- MVP：精簡物流與付款步驟、優化錯誤提示與回饋
```
